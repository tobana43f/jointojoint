### ゲーム概要

「**繋ぐ**」をテーマにした作品です。

プレイヤーは、キャラクターのコアと呼ばれる球体を操作し、
他のキャラクターや、ステージ上のオブジェクトに繋ぐことで、

## 🛠️技術的な見どころ

### 以前の自作エンジンをベースとしたシステム全体の刷新
本作では、前作「[BINARY-FIGHT](https://github.com/Tobana43F/BINARY-FIGHT)」で開発した自作エンジンをベースに、アーキテクチャから物理演算ロジックまでを全面的に刷新しています。  
単なる機能追加に留まらず、プログラムの保守性や汎用性を追求した設計へのアップグレードを行いました。

#### 描画システムの高度な抽象化と専門化
描画システムにおいては、基底クラスから派生した専門クラス（Polygon2D/3D, Text2D, Line）へ機能を細分化する設計を採用しました。  

描画ロジックを各コンポーネントにカプセル化することで、コードの重複を排除。  
これにより、テキスト描画や物理デバッグ用のライン表示といった新機能を、既存のコアシステムを破壊することなく柔軟に追加・拡張できる「部品化された描画エンジン」へと進化させました。

#### 疎結合な入力システム
入力系はDirectXのAPIを直接叩くのではなく、論理入力レイヤーを介する2レイヤー構造へと刷新しました。  
これにより、ゲームロジックがAPIに依存しない疎結合な設計となり、キーアサインの変更や移植への柔軟な対応が可能です。  

#### 独立したシーン管理によるメモリとフローの最適化
各シーン（タイトル、ゲーム本編、リザルト等）を独立したモジュールとして管理する構造を導入し、シーン遷移時の初期化・解放フローを共通化しました。  
これにより、メモリリークのリスクを最小限に抑えつつ、シーンの追加や変更が容易な、統制の取れたゲームフロー管理を実現しています。

### GJKアルゴリズムとミンコフスキー差を用いた衝突判定
物理演算の核となる当たり判定には、2つの凸集合の「ミンコフスキー差」を利用して最近接点を求めるGJKアルゴリズムをスクラッチで実装しました。（[ソースコード](https://github.com/Tobana43F/JoinToJoint/blob/554d73bd2a1990b9bcbdf92a8ef7ac8e29f3b888/Sources/RigidbodyPair.cpp#L95)）  
これにより、既存の物理ライブラリに頼ることなく、複雑な形状のオブジェクト同士でも「埋まり」や「すり抜け」のない精密な判定を数学的根拠に基づいて実現しています。  
アルゴリズムの解析から実装までを自ら行うことで、パズルゲームにおいて極めて重要となる物理的な整合性と、プレイヤーが感じる心地よい操作感を両立させました。


### 効率的な当たり判定パイプラインと負荷軽減の工夫
多数のオブジェクトが同時に存在するシーンでの実行速度を確保するため、判定処理をブロードフェーズとナローフェーズの2段階に分離しました。    
まずは計算負荷の低いAABBを用いて衝突の可能性を絞り込み、必要なペアに対してのみGJKアルゴリズムによる詳細判定を行うパイプラインを構築。  
この最適化により、処理コストと精度のトレードオフを高度に制御し、エンジン全体のパフォーマンスを最大化しています。

## ✨こだわり・アピールポイント

### ステージデータの外部ファイル化によるレベルデザイン作業の高速化
オブジェクト（足場、ギミック、キャラクター等）の配置情報を外部のCSVファイルとして管理し、  
ソースコードの修正やビルドを介さずに、Excel等の数値調整だけで即座にレイアウトを変更できる環境を構築しました。  

プログラマ以外のメンバーでも直接レベルデザインの試行錯誤が可能なワークフローを実現したことで、  
チーム内での作業分担を明確化するとともに、配置パターンの検証サイクルを大幅に短縮しています。  

また、データの外部化を前提とした汎用的なロード・生成マネージャーを設計しており、  
将来的なステージ追加にも柔軟かつ迅速に対応可能な、メンテナンス性の高いシステム構成となっています。

### 制約下で世界観を最大化するためのアート機能と表現手法
本プロジェクトのチームには専任のデザイナーがいなかったため、描き込まれた高精細なアセットの量産は現実的ではありませんでした。  

自分たちでも世界観を構築できる「シンプルかつ効果的な表現」を模索し、ゲーム『パタポン』のようなシルエット基調のデザイン方針を策定しました。  
具体的には、背景要素をシルエットとして表現し、多重スクロールを実装することで、少ないリソースでも奥行きと空気感のある世界観を実現しています。  
さらに、シルエット表現に深みを与えるため、各頂点に異なる色を設定する「頂点カラーグラデーション」を採用。  
背景の上部と下部で明度や色相を変化させることで、単一のテクスチャでも空の広がりや空気遠近法のような繊細な色彩表現を可能にしました。  

この工夫により、制作リソースの制約を技術的・企画的なアプローチで解決し、作品としての視覚的な完成度を引き上げることができました。

## 🔍自己レビュー

### 大きな値による初期化とマジックナンバーの利用
物理演算の探索処理（GJKアルゴリズム等）において、無限遠のシミュレートや最小値探索の初期値として「99999」といったマジックナンバーを直接記述してしまっている箇所があります。  
現在の視点で見れば、これらはコードの可読性と堅牢性の観点から明確に改善すべきポイントであると考えています。  

具体的には、標準ライブラリの定数である `FLT_MAX` や `INT_MAX` を利用したり、計算の意味に合わせた定数を明示的に定義して利用すべきです。  
そうすることで、値の意図が明確になるだけでなく、予期せぬオーバーフローの防止や、将来的なデータ精度の変更にも強い、より高度な信頼性を備えたコードベースを構築できると認識しています。

### ヘッダーファイルへの実装記述に関する検討
本プロジェクトでは、ゲッターやセッターを中心にヘッダーファイル内で直接実装を記述している箇所が多数存在しています。  

これはインライン展開による高速化を意図したものでしたが、  
本来「設計図（インターフェース）」であるべきヘッダーに「具体的な手順（実装）」が混在することで、クラスの役割が見通しにくくなるという弊害があります。  

また、実装の僅かな変更でも再コンパイルの連鎖を招き、ビルド時間を増大させる要因にもなります。  
現在は、テンプレート等の特殊な例を除き、実装を `.cpp` ファイルへ分離することで、インターフェースと実装の境界を明確にし、保守性とビルド効率を両立させる設計を重視しています。
